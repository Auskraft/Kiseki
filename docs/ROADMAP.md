## Статус реализации (обновлено 2026-06-14)

Весь CRUD медиа (форма, деталь, корзина), картинки-обложки, грид/список + поиск/фильтр/сортировка, теги, настройки и бэкап/восстановление на Я.Диск готовы, покрыты тестами (**85 зелёных**, `flutter analyze` чист), залиты в `main`. **Этапы B, C, D, E, F закрыты (M2–M6); этап A закрыт полностью (A4 — go_router).** Бэкап подтверждён на устройстве. **Инженерная полировка G сделана** (типизированные `Failure`→UI, реактивная проверка места, детект повреждения БД + экран восстановления, a11y-метки, скелетоны загрузки, граничные случаи эмодзи/CJK/длинных) + код-аудит и лаунчер-иконка. **Навигация/темы/IA настроек переработаны** (визуальная сессия): 4-вкладочная оболочка + 3 стиля нав-бара + кастом-иконки меню + **стиль кнопки добавления (FAB)** с общим «дышащим» градиентом (капсула+FAB) + 11 тем с экраном-пикером (смена мгновенная) + раздел «Документация»/privacy. sqlite3 вендорен локально (офлайн-сборка; на android либу бандлит native-asset, при сбое — `flutter clean`); БД открывается на устройстве. Остаётся до релиза — визуал/брендинг за владельцем: сплеш, мульти-картинки, пресеты-фильтров, first-run с restore, анимации/тактилка, ≥48 dp; наполнение вкладок Календарь/Картотека; доводка цветов 6 производных тем + текста privacy.

- **Навигация + темы + IA настроек (визуальная сессия, порт-адаптация из `blood_pressure_diary`):** оболочка `MainShell` из 4 вкладок (Главная=медиа · Календарь/Картотека — заглушки `ComingSoonPage` · Настройки) на ленивом `IndexedStack` + свайп; **3 стиля нав-бара** (Жидкий metaball / Классический / Капсула с анимир. градиентом), общий «эффект стекла» + степень + градиент-тумблер, экран-пикер с живым «телефон-превью» (`NavStyleCubit`); **кастомизация иконок вкладок** (`MenuIconsCubit`, пул ~36 иконок, свой пикер); **11 тем** (5 ручных хэндоффа + 6 производных через HSL-`_derive`) с экраном-пикером «Тема оформления» (сетка карточек-превью + режим); раздел **«Документация»** (политика конфиденциальности, черновик). Всё на токенах (ADR-17), go_router, get_it; **КАПС убран** из UI. +14 тестов (95 всего).

**Дальше:** визуал/брендинг за владельцем (сплеш, мульти-картинки, пресеты-фильтров, first-run с restore, анимации/тактилка, ≥48 dp); наполнить вкладки Календарь/Картотека; проверить на устройстве C2 (обложки), восстановление БД, новые темы и стили нав-бара. На бету: NFC-нормализация (`unorm_dart`), проактивная проверка места (нативный плагин).

## 0. Граница «ядро / домен» — фиксируется первой

Это контракт, вокруг которого строится весь план. Если он поедет — поедет всё. Фиксируется на M1, не по ходу.

**Ядро (`core/`) — пишется один раз, общее для всех доменов:**
UUID + `created_at`/`updated_at`/`deleted_at` (мягкое удаление + корзина) · оценка `rating 0–100` · теги (M:N) · `note` · `is_favorite` · `event_count` · картинки (конвейер 512/150 WebP, относит. пути, чистка, **таблица `images` 1:N**) · FTS5-поиск · фильтр/сортировка по общим полям · бэкап на Я.Диск · инфраструктура (Drift, миграции, DI, навигация, темизация).

**Домен (`features/<domain>/`) — добавляется на каждый новый тип:**
доменные поля и подмодели (для медиа: тип, `format`, сезон/серия, страна, статусы просмотра, причина паузы, даты) · доменные экраны · доменные фильтры.

Детали схемы и принятые решения — см. [TECH_DESIGN §2–§4](TECH_DESIGN.md#2-канонический-реестр-идентификаторов).

## 1. Итерация 1 (медиа): MVP vs nice-to-have

**MVP — must (без этого итерация не закрыта):**
- CRUD карточки медиа с мягким удалением и корзиной.
- Все поля из контекста: тип, `format`, название, оригинал, год, страна, теги, оценка, статус, `unfinished_reason` (+ детали в `note`), отзыв, картинка, даты, избранное, `event_count`.
- Различие фильм/сериал по `format` (у `episodic` — сезон/серия/всего/«остановился на»).
- Картинка: загрузка → WebP, два размера (thumb 150 + full 512), относит. путь, чистка файлов.
- Список + грид с превью (thumbnail), реактивно через `.watch()`.
- Поиск FTS5 (название + оригинал + заметка), фильтр (статус/тип/format/оценка/теги), сортировка (даты/оценка).
- Бэкап и восстановление (Replace-all) на Яндекс.Диск; миграции работают и протестированы хотя бы на одном переходе схемы.

**Nice-to-have (за пределы итерации, не блокирует релиз):** статистика/дашборд · кастомные представления оценки · массовые операции · автозаполнение из TMDb (нарушает offline-first) · drag-drop сортировка · приблизительные даты (флаг точности) · JSON-экспорт и Merge-LWW · авто-очистка корзины по сроку · виджеты/шаринг.

Правило: **must** = «бесполезно без этого» ИЛИ «дорого переделать» (схема, бэкап-формат, конвейер картинок, граница ядро/домен). Всё, что кладётся поверх стабильной схемы — nice-to-have.

## 3. Почему такой порядок снижает переделки

- **Каркас раньше фич.** Схема, UUID, граница ядро/домен, относит. пути, миграции — переделка каскадом ломает данные и весь код выше. Менять схему на боевых данных — самое дорогое в проекте.
- **CRUD до картинок.** Картинки подключаются к готовым точкам жизненного цикла записи; иначе чистку файлов переделывали бы под каждое изменение модели удаления.
- **Картинки до списков.** Списки сразу строятся на правильном (лёгком) thumbnail, а не переписываются после оптимизации.
- **Списки до поиска/фильтра.** Поиск — операция над существующим списком; его видно и проверяемо только на данных.
- **Поиск/фильтр до бэкапа.** Бэкап тестируется на наполненном наборе; он же последним вскрывает нехватку в схеме (что-то не хватает для мёржа) — лучше до косметики.
- **Полировка в конце** — дешевле всего, не влияет на данные.

Антипаттерн, который порядок предотвращает: «красивый UI на черновой схеме → поменяли схему → переписали UI+БД+сломали бэкап».

## 4. Будущие итерации: еда, рестораны, вейп

Ядро (этапы A, C, E, F + общая часть B) пишется один раз. Новый домен = доменная таблица + миграция + доменная форма + доменные фильтры.

**Переиспользуется во всех доменах:** UUID/даты/мягкое удаление/корзина · оценка 0–100 · теги/заметка/избранное · конвейер картинок (полностью) · FTS5/сортировка/фильтр по оценке·тегам · бэкап (новый домен попадает автоматически) · Drift/миграции/DI/навигация/темизация.

**Специфика:**
- **Еда** — калорийность/состав/срок годности; статусы «хочу попробовать/пробовал/готовлю»; нет сезон/серия.
- **Рестораны** — адрес/гео, ценовая категория, средний чек; **несколько фото** (место/блюда — отсюда таблица `images` 1:N в ядре, ADR-09); счётчик визитов = `event_count`.
- **Вейп** — крепость (мг/мл), PG/VG, объём, бренд/линейка; статусы «хочу/в ротации/закончилась/не зашла».

**Критерий успеха ядра:** добавить «вейп» = новая доменная таблица + миграция + форма + фильтры, **без изменений** ядро-схемы, конвейера картинок, бэкапа и базовых экранов. Единственная вынужденная правка ядра — строка в `@DriftDatabase(tables:[...])` (Drift статичен).

## 5. Сквозные темы (параллельно с M1)

- **Синхронизация устройств** — не живая; восстановление/мёрж через бэкап. Фундамент (UUID, `updated_at`, tombstones, относит. пути) — в схеме сразу; реализация Replace-all — этап F, Merge-LWW — будущая итерация (без смены формата).
- **Тесты** — с этапа A и растут с кодом: unit (репозитории, маппинг оценки, резолв путей), миграции (на каждую `schemaVersion` — обязательно), bloc (переходы), интеграция (бэкап→restore). DoD каждой задачи неявно включает «покрыто там, где есть логика».
- **i18n** — инфраструктура ключей (`gen-l10n` + `.arb`) с M1; в итерации 1 только русские ключи, переводы — на бете без смены архитектуры (ADR-15).
- **Темизация** — этап A; токенная система под десятки тем, новые экраны/домены берут значения только из токенов, не хардкод (ADR-17).
- **Аналитика/телеметрия** — нет (приватность, offline-first). Только локальный лог ошибок в debug (ADR-16).
