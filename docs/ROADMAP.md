## 0. Граница «ядро / домен» — фиксируется первой

Это контракт, вокруг которого строится весь план. Если он поедет — поедет всё. Фиксируется на M1, не по ходу.

**Ядро (`core/`) — пишется один раз, общее для всех доменов:**
UUID + `created_at`/`updated_at`/`deleted_at` (мягкое удаление + корзина) · оценка `rating 0–100` · теги (M:N) · `note` · `is_favorite` · `event_count` · картинки (конвейер 512/150 WebP, относит. пути, чистка, **таблица `images` 1:N**) · FTS5-поиск · фильтр/сортировка по общим полям · бэкап на Я.Диск · инфраструктура (Drift, миграции, DI, навигация, темизация).

**Домен (`features/<domain>/`) — добавляется на каждый новый тип:**
доменные поля и подмодели (для медиа: тип, `format`, сезон/серия, страна, статусы просмотра, причина паузы, даты) · доменные экраны · доменные фильтры.

Детали схемы и принятые решения — см. [TECH_DESIGN §2–§4](TECH_DESIGN.md#2-канонический-реестр-идентификаторов).

## 1. Итерация 1 (медиа): MVP vs nice-to-have

**MVP — must (без этого итерация не закрыта):**
- CRUD карточки медиа с мягким удалением и корзиной.
- Все поля из контекста: тип, `format`, название, оригинал, год, страна, теги, оценка, статус, `unfinished_reason` (+ детали в `note`), отзыв, картинка, даты, избранное, `event_count`.
- Различие фильм/сериал по `format` (у `episodic` — сезон/серия/всего/«остановился на»).
- Картинка: загрузка → WebP, два размера (thumb 150 + full 512), относит. путь, чистка файлов.
- Список + грид с превью (thumbnail), реактивно через `.watch()`.
- Поиск FTS5 (название + оригинал + заметка), фильтр (статус/тип/format/оценка/теги), сортировка (даты/оценка).
- Бэкап и восстановление (Replace-all) на Яндекс.Диск; миграции работают и протестированы хотя бы на одном переходе схемы.

**Nice-to-have (за пределы итерации, не блокирует релиз):** статистика/дашборд · кастомные представления оценки · массовые операции · автозаполнение из TMDb (нарушает offline-first) · drag-drop сортировка · приблизительные даты (флаг точности) · JSON-экспорт и Merge-LWW · авто-очистка корзины по сроку · виджеты/шаринг.

Правило: **must** = «бесполезно без этого» ИЛИ «дорого переделать» (схема, бэкап-формат, конвейер картинок, граница ядро/домен). Всё, что кладётся поверх стабильной схемы — nice-to-have.

## 2. Этапы с Definition of Done

Порядок строгий: дорогие-в-переделке решения принимаются раньше.

### Этап A. Каркас (фундамент)

**A1. Скелет и структура**
- [ ] Структура папок `core/` + `features/media/` (см. [TECH_DESIGN §5.1](TECH_DESIGN.md#51-дерево-папок-feature-first-ядро--домены-плагины)); демо-`CounterCubit` удалён.
- [ ] Слои зафиксированы: data (Drift) / domain (модели + интерфейсы) / presentation (bloc + UI).
- [ ] `flutter analyze` чист; приложение запускается с пустым home.

**A2. Drift: ядро-схема, FTS5, миграции**
- [ ] Подключены `drift`, `drift_flutter`, `sqlite3_flutter_libs`, `drift_dev`, `build_runner`.
- [ ] Каноническая схema: `catalog_items` (ядро) + `media_items` (1:1) + `images` (1:N) + `tags` + `item_tags` — точно по [TECH_DESIGN §4.2](TECH_DESIGN.md#42-sql-ddl-каноническое-описание). Имена/enum-коды — из [Реестра §2](TECH_DESIGN.md#2-канонический-реестр-идентификаторов).
- [ ] **Время — INTEGER мс UTC** через единый конвертер; `storeDateTimeAsText:false`; без `currentDateAndTime` в DEFAULT (ADR-04).
- [ ] `beforeOpen` включает `PRAGMA foreign_keys = ON`.
- [ ] FTS5 `catalog_fts` + триггеры **по `item_id` (UUID, не rowid)** + процедура rebuild (ADR-10).
- [ ] `MigrationStrategy`, `schemaVersion=1`, заготовка `onUpgrade`, тест открытия БД.
- [ ] `.watch()` пере-эмитит при изменении (проверено тестом на in-memory Drift).
- [ ] `build_runner` стабильно проходит.

**A3. DI**
- [ ] **get_it** + ручной `configureDependencies()` (ADR-13; `injectable` не тащим).
- [ ] БД/репозитории/сервисы — `lazySingleton`; bloc/cubit — factory.
- [ ] Репозитории за интерфейсами (для тестов/подмены).

**A4. Навигация**
- [ ] **go_router**; маршруты-заготовки: список → детали → редактор → корзина → настройки/бэкап.
- [ ] Deep-link на карточку по UUID; передача аргументов и back работают.

**A5. Темизация и i18n**
- [ ] Токенная тема (цвета/типографика/отступы/радиусы/тени) через `ThemeExtension`; реестр тем под **десятки тем** (у каждой light/dark), переключение в рантайме + «системная», выбор сохраняется. Ноль хардкода (ADR-17).
- [ ] i18n: `flutter gen-l10n` + `.arb`; все строки через ключи; в итерации 1 заполнены только русские значения, переводы — на бете (ADR-15).

**DoD A:** приложение открывает пустые экраны по роутеру, БД создаётся с ядро-схемой + FTS5, миграции/DI на месте, темы переключаются. Нулевая вертикаль без данных.

### Этап B. CRUD карточки

**B1. Доменные модели и репозиторий**
- [ ] `MediaEntry` + различие `single`/`episodic` (у `episodic` — сезон/серия/всего/«остановился на»).
- [ ] Enum статусов и `unfinished_reason` — коды из [Реестра](TECH_DESIGN.md#enum-коды-хранимые-строки--стабильны-навсегда); «жду серии» (`paused`+`waiting_episodes`) ≠ «заброшено».
- [ ] Даты nullable + точность (день/месяц/год): `started_at` (≠ `created_at`), `last_activity_at`, `finished_at` — пользователь может задать ориентировочно.
- [ ] Репозиторий CRUD: create (UUID + `created_at`/`updated_at`), read, update (двигает `updated_at`), softDelete (`deleted_at`), restore, purge — **транзакционно**, ядро→домен ([TECH_DESIGN §4.5](TECH_DESIGN.md#45-транзакционные-инварианты-обязательны)).

**B2. Экран создания/редактирования**
- [ ] Прогрессивная форма; поля серий — только при `format=episodic`; `unfinished_reason` — только при `paused`/`dropped` (реактивно).
- [ ] Валидация: `title` обязателен, год/оценка в диапазонах; оценка — 100-балльная (слайдер/число 0–100).
- [ ] Сохранение через bloc → репозиторий; ошибки видны пользователю.

**B3. Детальная карточка**
- [ ] Все поля, оценка в выбранном представлении, статус + быстрые переходы, даты, теги, заметка, избранное, `event_count` (+1).
- [ ] Действия: редактировать, в избранное, мягко удалить.

**B4. Мягкое удаление и корзина**
- [ ] softDelete убирает из основных списков; экран корзины + восстановление; purge удаляет запись и (этап C) файлы картинок.

**DoD B:** можно завести фильм и сериал со всеми полями, отредактировать, удалить в корзину, восстановить. Списки черновые, картинок ещё нет (намеренно).

### Этап C. Конвейер картинок

**C1. Сервис обработки**
- [ ] Выбор (галерея/камера); приведение к ≤512×512, WebP; **два размера** thumb 150 / full 512; параметры из [TECH_DESIGN §7.1](TECH_DESIGN.md#71-параметры-максимум-512512).
- [ ] Файлы в `<AppSupport>/media/full|thumb/<uuid>.webp` через `.tmp`+rename; в БД (`images`) — только UUID; абсолютный путь резолвится в рантайме.
- [ ] Unit-тест инварианта «в БД нет абсолютного пути».

**C2. Интеграция в CRUD + чистка**
- [ ] Карточка — `full`, списки — `thumb` (обложка = `MIN(position)`). Медиа допускает несколько картинок (обычно одна, изредка до 3). Замена → новый UUID + `imageCache.evict` (без «старого кадра»).
- [ ] purge удаляет файлы; orphan sweeper; плейсхолдер при отсутствии/потере файла (без краша).

**DoD C:** картинка прикрепляется, сжата, в двух размерах, путь относительный; при удалении карточки файлы исчезают; нет сирот.

**DoD D:** карточки видны списком и гридом на лёгких thumbnail, всё реактивно.

### Этап E. Поиск / фильтр / сортировка

**E1. Поиск (FTS5)**
- [ ] По `title`/`original_title`/`note`; реактивно, дебаунс ~250–300 мс.
- [ ] Триггеры FTS отрабатывают на create/update/delete (тест на отсутствие рассинхрона). Теги — реляционным фильтром, не через FTS.

**E2. Фильтр и сортировка**

- [ ] Сортировки: даты, оценка (null в хвост), `created_at`/`updated_at`, название, год; выбор живёт в сессии.

**DoD E:** нужная карточка находится по названию и отсеивается фильтрами; сортировка корректна.

### Этап F. Бэкап

**F1. Снимок и формат**
- [ ] Архив `.kiseki` = `VACUUM INTO snapshot.sqlite` + `images/` + `manifest.json` (`format_version` + `schema_version` + sha256). Порядок файл→коммит/коммит→удаление соблюдён ([TECH_DESIGN §8](TECH_DESIGN.md#8-бэкап-и-восстановление-яндексдиск)).

**F2. Яндекс.Диск + восстановление**
- [ ] OAuth (Authorization Code + PKCE), токен в Keychain/Keystore; REST API; заливка `*.partial`→rename; ротация N последних; индикация последнего успешного бэкапа.
- [ ] **Restore = Replace-all** отдельным флоу: подтверждение + safety-snapshot → teardown bloc/БД → атомарная замена → пересоздание синглтона → ре-навигация. Drift прогоняет миграции.
- [ ] Сценарий «новое устройство»: чистая установка → восстановление → данные и картинки на месте; tombstones не «воскресают».

**DoD F:** бэкап уходит на Я.Диск и восстанавливается на другом устройстве. Валидирует UUID + относительные пути + миграции на практике.

### Этап G. Полировка и релиз

**G1. Доведение**
- [ ] Управление тегами (переименование/слияние дублей/удаление/цвет); подтверждение деструктива; первый запуск с опцией restore.
- [ ] a11y-проход (Semantics, ≥48 dp, оценка текстом не только цветом, `textScale` 1.3–2.0); типизированные `Failure` → действия в UI; проверка свободного места.
- [ ] Граничные случаи: длинные названия (NFC, ellipsis), нет картинки, пустая БД, большой объём, эмодзи/CJK.
- [ ] Анимации/тактильная отдача; иконка/сплеш. Финальные тесты зелёные, `flutter analyze` чист; ручная проверка миграции на реальном переходе схемы.

**DoD G и всей итерации 1:** все must из §1 закрыты, бэкап-цикл между устройствами работает, миграции проверены.

## 3. Почему такой порядок снижает переделки

- **Каркас раньше фич.** Схема, UUID, граница ядро/домен, относит. пути, миграции — переделка каскадом ломает данные и весь код выше. Менять схему на боевых данных — самое дорогое в проекте.
- **CRUD до картинок.** Картинки подключаются к готовым точкам жизненного цикла записи; иначе чистку файлов переделывали бы под каждое изменение модели удаления.
- **Картинки до списков.** Списки сразу строятся на правильном (лёгком) thumbnail, а не переписываются после оптимизации.
- **Списки до поиска/фильтра.** Поиск — операция над существующим списком; его видно и проверяемо только на данных.
- **Поиск/фильтр до бэкапа.** Бэкап тестируется на наполненном наборе; он же последним вскрывает нехватку в схеме (что-то не хватает для мёржа) — лучше до косметики.
- **Полировка в конце** — дешевле всего, не влияет на данные.

Антипаттерн, который порядок предотвращает: «красивый UI на черновой схеме → поменяли схему → переписали UI+БД+сломали бэкап».

## 4. Будущие итерации: еда, рестораны, вейп

Ядро (этапы A, C, E, F + общая часть B) пишется один раз. Новый домен = доменная таблица + миграция + доменная форма + доменные фильтры.

**Переиспользуется во всех доменах:** UUID/даты/мягкое удаление/корзина · оценка 0–100 · теги/заметка/избранное · конвейер картинок (полностью) · FTS5/сортировка/фильтр по оценке·тегам · бэкап (новый домен попадает автоматически) · Drift/миграции/DI/навигация/темизация.

**Специфика:**
- **Еда** — калорийность/состав/срок годности; статусы «хочу попробовать/пробовал/готовлю»; нет сезон/серия.
- **Рестораны** — адрес/гео, ценовая категория, средний чек; **несколько фото** (место/блюда — отсюда таблица `images` 1:N в ядре, ADR-09); счётчик визитов = `event_count`.
- **Вейп** — крепость (мг/мл), PG/VG, объём, бренд/линейка; статусы «хочу/в ротации/закончилась/не зашла».

**Критерий успеха ядра:** добавить «вейп» = новая доменная таблица + миграция + форма + фильтры, **без изменений** ядро-схемы, конвейера картинок, бэкапа и базовых экранов. Единственная вынужденная правка ядра — строка в `@DriftDatabase(tables:[...])` (Drift статичен).

## 5. Сквозные темы (параллельно с M1)

- **Синхронизация устройств** — не живая; восстановление/мёрж через бэкап. Фундамент (UUID, `updated_at`, tombstones, относит. пути) — в схеме сразу; реализация Replace-all — этап F, Merge-LWW — будущая итерация (без смены формата).
- **Тесты** — с этапа A и растут с кодом: unit (репозитории, маппинг оценки, резолв путей), миграции (на каждую `schemaVersion` — обязательно), bloc (переходы), интеграция (бэкап→restore). DoD каждой задачи неявно включает «покрыто там, где есть логика».
- **i18n** — инфраструктура ключей (`gen-l10n` + `.arb`) с M1; в итерации 1 только русские ключи, переводы — на бете без смены архитектуры (ADR-15).
- **Темизация** — этап A; токенная система под десятки тем, новые экраны/домены берут значения только из токенов, не хардкод (ADR-17).
- **Аналитика/телеметрия** — нет (приватность, offline-first). Только локальный лог ошибок в debug (ADR-16).

## 6. Вехи

- **M1 — Фундамент.** Этап A: структура, Drift+ядро-схема+FTS5+миграции, DI, навигация, темизация, каркас тестов. Граница ядро/домен зафиксирована.
- **M2 — Запись живёт.** Этап B: CRUD со всеми полями, фильм/сериал, корзина.
- **M3 — Картинки.** Этап C: 512/150 WebP, относит. пути, чистка.
